# AIKI Content Generation Rules

이 문서는 AIKI 뉴스, 위키, 프로젝트 쇼케이스 생성 규칙의 기준 문서다.

## 핵심 원칙

1. 스크립트는 글을 쓰지 않는다.
2. 스크립트는 재료를 준비하고 구조를 검증한다.
3. 실제 본문, 관련 용어 설명, 팩트체크 설명문은 LLM이 쓴다.
4. 글쓰기 규칙의 단일 출처는 `docs/content-guide-common.md` 와 타입별 `docs/content-guide-*.md` 다.
5. 프로젝트 쇼케이스의 사용자 노출 문장은 한국어로 쓰고, 초보자가 막히는 용어는 인라인 설명으로 푼다.

## 역할 분리

### 스크립트가 맡는 일

- 출처 수집
- 원문 제목, URL, 메타데이터 정규화
- 숫자, 이름, 버전, 회사명 같은 검증 후보 추출
- `readerValue` 같은 내부 방향성 필드 준비
- frontmatter 스키마 정리
- 배포 전 검증

### LLM이 맡는 일

- 제목
- 요약
- 본문 전체
- 관련 용어 설명
- 모델 프로필 설명문
- 팩트체크 요약과 상세 설명문
- 프로젝트 쇼케이스의 한국어 문구와 용어 설명

LLM은 톤 규칙을 이 문서에서 따로 정의하지 말고, 항상 applicable `content-guide`를 기준으로 쓴다.

## 프로젝트 쇼케이스 규칙

1. `lang: ko` 프로젝트 페이지와 쇼케이스에서 사용자에게 보이는 문장형 카피는 한국어로 쓴다.
2. 프로젝트 쇼케이스는 초보자를 기본 독자로 가정하고, 용어를 아는 사람끼리만 통하는 설명을 그대로 두지 않는다.
3. 어려운 용어는 섹션 말미 용어집으로 미루지 말고, 등장 위치에서 바로 이해할 수 있게 `?` 아이콘이 붙은 인라인 툴팁이나 동급의 설명 UI로 푼다.
4. 프로젝트 본문에 별도 `용어 가이드` 섹션을 두는 대신, 본문과 쇼케이스 안에서 필요한 만큼 바로 설명한다.
5. 영문 API 이름, 저장소 경로, CLI 명령, 고유명사는 원문 그대로 둘 수 있지만, 카드 제목, 섹션 리드, 설명 문장, 상태 문구는 한국어로 정리한다.

## 위키 작성 규칙

1. 카테고리 설명은 짧게 끝낸다.
2. 본문 중심은 해당 항목의 특징과 차별점이어야 한다.
3. 비슷한 카테고리의 다른 항목과 무엇이 갈리는지 먼저 설명한다.
4. "이게 어떤 카테고리인가"보다 "독자가 이 항목을 왜 구분해야 하는가"를 우선한다.
5. 같은 카테고리 문서끼리 문장 골격이 반복되면 실패로 본다.

### 금지 패턴

- 카테고리 정의만 길게 반복하는 문장
- 제품이나 기술 이름만 바꿔도 그대로 쓸 수 있는 설명
- "이것도 벡터 DB다", "이것도 프레임워크다" 수준의 분류 중심 서술
- 실제 차별점 없이 관련 용어만 바꾸는 방식

## 팩트체크 규칙

1. 팩트체크는 요약 한 줄로 끝내지 않는다.
2. `원문 대조`, `교차검증`, `숫자·명칭 검증`, `비판적 검증`을 모두 상세하게 적는다.
3. 각 항목에는 실제로 무엇을 확인했는지가 보여야 한다.
4. 숫자, 이름, 버전, 회사명, 제품 범위, 출시 채널처럼 독자가 오해하기 쉬운 부분을 구체적으로 남긴다.
5. 팩트체크 문장도 applicable `content-guide`와 같은 글쓰기 규칙을 써야 한다.
6. 팩트체크 안에서 같은 문장을 반복하지 않는다.
7. `봤다`, `맞춰봤다`, `걸렀다` 같은 메모체를 그대로 두지 않는다.

### 팩트체크에서 반드시 보여야 하는 것

- 어떤 원문을 기준으로 맞췄는지
- 어떤 다른 출처와 교차검증했는지
- 어떤 숫자와 명칭을 따로 확인했는지
- 어떤 오해 가능성을 의심했고 왜 걸러냈는지

## `readerValue` 규칙

1. `readerValue`는 사용자 화면에 직접 노출하지 않는다.
2. `readerValue`는 문서 방향을 잡는 내부 입력이다.
3. 본문, 관련 용어, 팩트체크는 모두 `readerValue`가 가리키는 독자 문제를 중심으로 작성한다.
4. `readerValue`가 실제 본문에 반영되지 않으면 실패로 본다.

## 재발 방지 원칙

1. 스크립트 템플릿이 본문 문장을 직접 완성하지 않게 유지한다.
2. 스크립트는 "무슨 재료가 필요한지"까지만 정하고, 최종 서술은 LLM이 맡는다.
3. 새 생성기를 만들 때도 먼저 이 문서 기준으로 역할을 나눈다.
4. 검증기는 applicable `content-guide` 기준 위반, 차별점, 팩트체크 상세도, 독자 문제 반영 여부를 모두 검사해야 한다.
5. 같은 문제가 반복되면 템플릿 문장만 고치지 말고 생성 파이프라인의 역할 분리부터 다시 점검한다.
6. 딱딱한 잔여 문구 판정도 개별 문서가 아니라 applicable `content-guide` 기준으로 처리한다.
