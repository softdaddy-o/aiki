# AIKI Wiki Page Validation Guide

이 문서는 AIKI 위키 페이지를 검수할 때 쓰는 기준 문서다.

## 통과 조건

모든 위키 페이지는 아래 질문에 답해야 한다.

1. 이 용어가 무엇인지 명확한가
2. 어느 계층에 놓이는지 보이는가
3. 비슷한 다른 것과 차이가 드러나는가
4. 무엇을 할 수 있는지 실무 감각으로 설명하는가
5. 독자가 왜 이 용어를 알아야 하는지 보이는가

## 필수 통과 항목

### 1. 정체성과 카테고리

- 첫 문단 또는 `한 줄 정의`에서 카테고리와 역할이 동시에 보여야 한다.
- "도구다", "기법이다", "프레임워크다"만 말하면 실패다.

### 2. 차별점

- 최소 두 개 이상의 구분 축이 있어야 한다.
- 구분 축 예시:
  - 인터페이스
  - 운영 방식
  - 배포 환경
  - 성능 특성
  - 데이터 흐름
  - 표준화 범위

### 3. 실제 사용 장면

- 최소 두 개 이상의 실제 사용 장면이 떠올라야 한다.
- 제품 페이지라면 "어디에 붙여 쓰는가"가 보여야 한다.
- 기술 페이지라면 "어느 단계에서 개입하는가"가 보여야 한다.

### 4. 관련 용어 품질

- 모든 관련 용어는 현재 페이지 기준 비교 축을 포함해야 한다.
- "같이 보면 좋다" 같은 일반 문구는 실패다.

### 5. 팩트체크 깊이

- `원문 대조`, `교차검증`, `숫자·명칭 검증`, `비판적 검증` 네 항목이 모두 있어야 한다.
- 각 항목은 최소 세 줄 이상의 구체 검증 내역을 포함해야 한다.
- 숫자, 버전, 회사명, 엔드포인트, 스펙, 범위처럼 오해하기 쉬운 포인트를 하나 이상 직접 확인해야 한다.

## 실패 조건

아래 중 하나라도 걸리면 실패다.

1. 카테고리 설명이 본문 대부분을 차지한다.
2. 다른 제품 이름으로 바꿔도 그대로 맞는 문장이 많다.
3. 관련 용어 설명이 generic하다.
4. 팩트체크가 추상적이다.
5. 제품, 모델, 프로토콜, 기법 분류가 틀렸다.
6. `readerValue`와 본문 방향이 맞지 않는다.

## 경고 조건

아래는 즉시 실패는 아니지만 수정 권고다.

1. 일반론이 너무 길다.
2. 고유명사나 실제 기능명이 너무 적다.
3. 비교 축이 하나뿐이다.
4. 실무 장면보다 정의 비중이 높다.

## 검증 체크리스트

### 이해도

- 이 페이지 하나로 "이게 뭔지" 설명할 수 있나
- 카테고리와 위치를 한 문장으로 말할 수 있나

### 구분 가능성

- 비슷한 다른 것과 최소 두 개 이상의 차이를 말할 수 있나
- 왜 헷갈리는지, 그래서 어디를 봐야 하는지 적혀 있나

### 활용도

- 이걸 어디에 붙여 쓰는지 떠오르나
- 무엇을 할 수 있고 무엇을 기대하면 안 되는지 드러나나

### 팩트체크

- 공식 원문 기준이 명시돼 있나
- 교차검증 소스가 둘 이상 있나
- 숫자나 이름 검증이 실제로 들어 있나
- 과장되거나 오해할 수 있는 표현을 직접 걷어냈나

## 좋은 예와 나쁜 예

좋은 예:

- "ComfyUI는 모델이 아니라 노드 그래프로 생성 파이프라인을 설계하고 실행하는 워크플로 앱이다."
- "Qdrant는 벡터를 저장하는 것에 그치지 않고, 필터링과 hybrid search까지 포함한 검색 계층으로 보는 편이 맞다."
- "MCP는 함수 호출 그 자체가 아니라 도구를 발견하고 연결하는 규격을 표준화한다."

나쁜 예:

- "ComfyUI는 AI 도구다."
- "Qdrant는 벡터 DB다."
- "MCP는 프레임워크다."

## 배포 전 최종 질문

1. 이 페이지는 용어 설명 페이지로서 제 역할을 하는가
2. 독자가 전체 그림, 위치, 차이, 활용을 모두 잡을 수 있는가
3. 같은 카테고리 다른 페이지와 문장 골격이 겹치지 않는가
4. 팩트체크가 실제 검증 로그처럼 읽히는가
