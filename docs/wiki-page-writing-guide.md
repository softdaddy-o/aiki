# AIKI Wiki Page Writing Guide

이 문서는 AIKI 위키 페이지를 LLM이 작성할 때 참조하는 기준 문서다.

## 목표

모든 위키 페이지는 독자가 "이 용어가 정확히 무엇인지" 이해하게 만들어야 한다.  
좋은 위키 페이지는 아래 질문에 답해야 한다.

1. 이 용어는 AI 스택 전체 그림에서 어디에 놓이는가
2. 어떤 카테고리에 속하는가
3. 비슷한 다른 것과 무엇이 다르고, 왜 구분해야 하는가
4. 실제로 무엇을 할 수 있게 해주는가
5. 언제 쓰고 언제 과하게 기대하면 안 되는가

## 핵심 원칙

1. 카테고리 설명은 짧게 끝낸다.
2. 본문 중심은 "이 항목만의 특징과 차별점"이어야 한다.
3. 독자가 실제로 헷갈리는 비교 포인트를 먼저 푼다.
4. `readerValue`가 가리키는 독자 문제를 본문 전체의 축으로 삼는다.
5. 다른 제품이나 기술 이름만 바꿔도 그대로 들어맞는 문장은 실패다.

## 기본 구조

모든 위키 페이지는 아래 구조를 기본으로 쓴다.

### `## 한 줄 정의`

- 카테고리와 역할을 한 문장으로 정리한다.
- "무엇을 위한 무엇"인지 바로 보여준다.
- 분류만 말하지 말고 기능적 위치를 같이 넣는다.

### `## 어떻게 작동하나` 또는 `## 실제로 무엇을 하나`

- 개념 페이지라면 작동 원리를 설명한다.
- 제품이나 도구 페이지라면 실제로 무엇을 해주는지 설명한다.
- 사용자가 어디에 붙여 쓰는지, 입력과 출력이 무엇인지, 어떤 계층에서 동작하는지 보여준다.

### `## 왜 중요한가`

- 도입 이유를 쓴다.
- 독자가 이 용어를 몰라서 어떤 판단을 틀리게 되는지 설명한다.
- 실무에서 직접 체감되는 사용 맥락을 넣는다.

### `## 관련 용어`

- 단순 나열 금지.
- 각 항목은 "왜 같이 봐야 하는지"를 현재 페이지 기준으로 설명한다.
- 비교 축은 운영 방식, 성능 특성, 데이터 흐름, 인터페이스, 비용 구조 중 하나 이상이어야 한다.

## 카테고리별 작성 포인트

### `tool`

- "어떤 작업을 실제로 해주는 제품/도구인가"를 먼저 쓴다.
- 실행 환경, 인터페이스, 운영 방식, 배포 감각이 드러나야 한다.
- 같은 카테고리 다른 도구와 갈리는 기준을 최소 두 개 넣는다.

### `framework`

- 코드를 어떻게 조직하고 연결하는지 설명한다.
- 단순 기능 목록보다 구조적 장점을 먼저 쓴다.
- "무엇을 표준화하는가", "무엇을 감춘는가", "어디까지 책임지는가"를 구분한다.

### `technique`

- 모델 자체가 아니라 문제 해결 방식이라는 점을 분명히 한다.
- 어떤 단계에서 개입하는지, 정확도/비용/지연 중 무엇에 영향을 주는지 적는다.
- RAG, grounding, distillation처럼 비슷한 용어와 경계를 반드시 짚는다.

### `concept`

- 추상 개념을 운영 판단에 연결해야 한다.
- 정의만 말하지 말고 어디에 쓰이는지, 왜 계속 등장하는지를 쓴다.

### `model`

- 모델 계열 페이지와 버전 페이지를 구분한다.
- 계열 페이지는 포지션과 차이를 설명한다.
- 버전 페이지는 실제 운영 정보, 접근 채널, 입력/출력 범위, 가격 또는 제한 조건을 함께 설명한다.

## 관련 용어 작성 규칙

좋은 관련 용어 설명은 이런 형태다.

- `[Stable Diffusion]` - ComfyUI에서 가장 자주 연결되는 모델 계열이라서, UI가 아니라 실제 생성 파이프라인이라는 점을 이해하는 데 도움이 된다.
- `[Pinecone]` - Qdrant와 같은 벡터 검색 계층이지만 관리형 서비스라는 점에서 운영 제어권과 책임 범위가 갈린다.

나쁜 관련 용어 설명은 이런 형태다.

- "같이 보면 좋다"
- "맥락을 같이 잡아 준다"
- "연관이 있다"

## 팩트체크 서술 규칙

팩트체크도 본문과 같은 톤으로 쓴다. 보고서 메모처럼 끊지 말고, 독자가 "무엇을 확인했는지" 바로 읽히게 쓴다.

반드시 아래 네 묶음을 모두 채운다.

1. `원문 대조`
2. `교차검증`
3. `숫자·명칭 검증`
4. `비판적 검증`

각 묶음에는 최소 세 개 이상의 구체 항목이 있어야 한다.

좋은 항목 예시:

- "공식 워크플로 문서는 workflow를 connected graph of nodes라고 설명한다."
- "Ollama API 소개 문서는 기본 로컬 엔드포인트를 `http://localhost:11434/api`로 적는다."
- "Qdrant 공식 사이트는 dense, sparse, multivector retrieval을 한 제품 범위 안에 묶어 소개한다."

나쁜 항목 예시:

- "공식 문서와 대체로 일치한다."
- "숫자와 이름을 확인했다."
- "오해 소지가 적다."

## 문체 규칙

1. 친절하게 설명하되 과장하지 않는다.
2. "쉽게 말하면", "실무에서는", "여기서 중요한 건" 같은 연결 문장은 쓸 수 있지만 남발하지 않는다.
3. 정의만 반복하지 말고 판단 기준을 준다.
4. 독자가 이 용어를 다른 것과 구분하게 만드는 문장을 먼저 배치한다.
5. 한 페이지 안에서 같은 문장 골격을 반복하지 않는다.

## 최종 점검 질문

배포 전에 아래 질문에 모두 "예"라고 답할 수 있어야 한다.

1. 이 페이지 하나만 읽어도 그 용어가 무엇인지 감이 오나
2. AI 스택 전체 그림에서 어디에 놓이는지 알 수 있나
3. 비슷한 다른 것과 어디서 갈리는지 알 수 있나
4. 실제로 무엇을 할 수 있는지 떠오르나
5. 관련 용어 설명이 단순 링크가 아니라 비교 기준을 주나
6. 팩트체크가 추상 요약이 아니라 실제 검증 내역을 보여주나
