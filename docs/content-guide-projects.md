# AIKI Content Guide — Projects

> **Guide ID**: `projects`
> **Version**: `3.0.0`
> **Last Updated**: 2026-04-21
> **Depends on**: `content-guide-common.md` v1.0.0+
> **Applies to**: `src/content/projects/ko/*.md` frontmatter/body + 사용자에게 보이는 showcase copy

이 문서는 AIKI 프로젝트 페이지 생성에 적용되는 텍스트 규칙이다.
공통 규칙은 `content-guide-common.md`를 참조한다.
레이아웃, 카드 배치, responsive 규칙은 UI/UX 문서가 다루고, 이 문서는 문장과 판단 흐름만 다룬다.

---

## 1. 프로젝트 페이지 목표

독자가 저장소 소개를 읽는 데서 끝나지 않고, "지금 바로 테스트할지", "보류할지", "어떤 워크플로로 검증할지"를 빠르게 가를 수 있어야 한다.

좋은 프로젝트 페이지의 기준:
1. 첫 구간에서 프로젝트가 뭘 하는지와 어디서 빛나는지가 바로 보인다
2. 실제 실행, 화면, 입력/출력, 운영 감각이 텍스트에 들어 있다
3. 독자가 붙여 볼 대상과 피해야 할 조건이 같이 보인다
4. 카드, 본문, factCheck가 같은 문체와 같은 판단축을 유지한다

---

## 2. 적용 범위

- frontmatter `summary`, `readerValue`
- markdown body
- showcase 카드 제목, 카드 본문, 섹션 리드, 캡션, 상태 문구
- factCheck summary / items / findings

프로젝트 페이지에서 사용자에게 보이는 문장형 카피는 전부 이 가이드를 따른다.

---

## 3. 프로젝트 카피 규칙

### 3-1. 공통 톤 기준을 그대로 따른다

프로젝트도 `content-guide-common.md`의 구어체 혼합 기준과 금지 패턴을 그대로 따른다.
설명체나 보고서체로만 밀어붙이지 않는다.

### 3-2. 첫 문단과 summary는 프로젝트 판단으로 시작한다

- "이 페이지는", "이 글은", "이 쇼케이스는"처럼 페이지 자체를 소개하는 문장으로 열지 않는다
- 프로젝트, 사용처, 제약, 데모 결과 중 하나로 바로 시작한다
- `readerValue`도 페이지 효용 설명이 아니라 독자의 실제 선택지를 써야 한다

**나쁜 예**
> 이 페이지는 PocketBase를 빠르게 판단하기 위한 쇼케이스다.

**좋은 예**
> PocketBase는 SQLite와 admin UI를 한 파일에 묶어서 내부툴 백엔드를 아주 짧게 띄울 때 힘이 세다.

### 3-3. 비교 문구만 남기고 끝내지 않는다

`X보다 Y에 가깝다`, `A쪽이다`, `B에 맞다` 같은 비교 문구는 보조 설명으로만 쓴다.
이 표현이 나오면 바로 뒤에 최소 하나는 붙어야 한다:

- 바로 해볼 테스트
- 운영상 갈리는 기준
- 붙이면 이득인 워크플로
- 피해야 할 조건

비교 문구만 연달아 쌓이면 FAIL이다.

### 3-4. 카드 제목은 라벨형으로 짧게 쓴다

- 카드 제목과 섹션 헤더는 산문 문장으로 쓰지 않는다
- 제목은 명사구, 판단 축, 기능 묶음처럼 짧게 둔다
- 설명은 제목 아래 본문에서 푼다

**좋은 예**
- `자동 Admin CRUD`
- `내장 확장 경로`
- `실행 확인 경로`

**나쁜 예**
- `Admin CRUD는 collection만 잡으면 바로 운영 가능한 수준까지 나온다`
- `repo sample이 설명용 문서가 아니라 실제 바이너리와 거의 붙어 있다`

### 3-5. 본문과 factCheck는 평평한 종결어미 반복을 피한다

`~된다`, `~있다`, `~갈린다`, `~맞다` 같은 설명체 종결어미가 key section과 factCheck에 반복되면 텍스트가 판정문처럼 굳어 버린다.
구체 동작, 수치, 관찰, 다음 행동을 섞어 리듬을 깨야 한다.

### 3-6. 데모가 있으면 입력과 출력을 같이 적는다

프로젝트 쇼케이스는 내부 UI 설명보다 실제 입력과 실제 출력이 먼저 보여야 한다.
화면, 명령, 응답, 저장 결과, 메모리 수치처럼 독자가 그대로 따라 볼 수 있는 근거를 우선한다.

---

## 4. 리뷰 체크포인트

5인 리뷰와 수동 리뷰에서 아래를 같이 본다.

1. common 톤 규칙 위반이 없는가
2. 첫 구간이 페이지 소개가 아니라 프로젝트 판단으로 열리는가
3. 비교 문구가 concrete go/no-go 없이 누적되지 않는가
4. 카드 제목과 헤더가 산문 문장이 아닌가
5. factCheck까지 같은 문체와 판단축을 유지하는가
