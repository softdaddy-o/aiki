## 한 줄 정의
LlamaIndex는 문서, 데이터베이스, API 같은 외부 정보를 LLM이 질의하고 활용할 수 있게 연결하는 프레임워크야. 모델 파라미터를 바꾸는 도구라기보다, 답변에 넣을 근거 데이터를 준비하고 찾는 층을 맡아.
이 프레임워크는 텍스트 조각화, 메타데이터 관리, 인덱싱, 검색, 질의 엔진 구성을 한 흐름으로 묶어 줘. 그래서 초점은 모델 성능 자체보다 검색과 외부 지식 연결 방식을 어떻게 설계하느냐에 맞춰져 있어.
## 어떻게 작동하나
데이터를 먼저 읽어 와서 검색하기 좋은 단위로 나누고, 필요하면 메타데이터를 붙인 뒤 인덱스나 벡터 저장소에 연결해 둬. 이 준비 단계가 잘 되어 있어야 질문이 들어왔을 때 관련 조각을 안정적으로 다시 찾을 수 있어.
사용자가 질문하면 리트리버나 쿼리 엔진이 관련 문서 조각을 골라내고, 그 결과를 LLM 프롬프트에 넘겨 답을 만들게 해. 여기에 재정렬, 필터링, 도구 호출, 에이전트 흐름을 얹을 수 있어서 단순 문서 검색부터 문서 기반 작업 자동화까지 확장할 수 있어.
## 왜 중요한가
실제 서비스에서는 모델이 학습 시점에 모르던 사내 문서, 최신 정책, 제품 매뉴얼, 고객별 데이터를 다뤄야 하는 경우가 많아. 이때 답변 품질은 어떤 모델을 쓰느냐 못지않게 어떤 자료를 가져오고 어떤 근거를 골랐느냐에 크게 흔들려.
LlamaIndex가 중요한 이유는 이 검색·연결 층을 직접 다룰 수 있게 해 주기 때문이야. 그래서 이 이름이 등장하면 새 언어모델 이야기라기보다, RAG나 문서 에이전트를 만들 때 외부 지식을 붙이는 구조를 어떻게 짜는지와 더 가깝게 읽는 편이 정확해.
## 주의해서 볼 점
LlamaIndex를 쓴다고 검색 정확도가 자동으로 올라가지는 않아. 청킹 방식, 임베딩 선택, 메타데이터 설계, 재정렬 전략, 저장소 구성이 어긋나면 프레임워크가 있어도 결과는 쉽게 흔들려.
이름이 가리키는 범위도 구분해서 봐야 해. 오픈소스 맥락에서는 데이터 연결과 질의 흐름을 다루는 프레임워크를 뜻하는 경우가 많지만, 공식 채널에서는 문서 파싱이나 워크플로 제품군까지 함께 묶여 보여서 제품 라인업과 핵심 프레임워크를 혼동하면 해석이 빗나갈 수 있어.
## 관련 용어
- [LangChain](/ko/wiki/langchain/) — LangChain은 체인, 에이전트, 애플리케이션 조립 전반을 넓게 다루는 편이야. LlamaIndex는 그중에서도 데이터 적재, 인덱싱, 검색, 질의 계층에 더 무게가 실려 있어서 "외부 지식을 어떻게 붙일까"라는 문제를 볼 때 비교점이 선명해.
- [RAG](/ko/wiki/rag/) — RAG는 검색 결과를 생성 단계에 넣어 답하게 하는 방법 자체를 말해. LlamaIndex는 그 방법을 실제 시스템으로 구현할 때 쓰는 프레임워크라서, 개념과 구현 도구를 구분하는 데 필요해.
- [Tool Use](/ko/wiki/tool-use/) — tool use는 모델이 검색기나 계산기 같은 외부 기능을 호출하는 동작을 뜻해. LlamaIndex는 그 도구 호출 흐름 안에 문서 검색과 질의 엔진을 엮어 넣는 데 자주 쓰여서, 호출 행위 자체와 데이터 연결 레이어를 나눠 보는 비교가 중요해.
- [Embedding](/ko/wiki/embedding/) — embedding은 텍스트를 수치 벡터로 바꿔 유사도를 계산하게 하는 표현 방식이야. LlamaIndex는 그 벡터를 만드는 단계만이 아니라 저장, 검색, 후처리, 응답 연결까지 감싸는 상위 프레임워크라는 점에서 역할 범위가 다르다.
