## 한 줄 정의
GitHub Copilot은 에디터 안에서 현재 코드와 주변 문맥을 보고 다음 줄, 함수 초안, 설명을 제안하는 GitHub의 AI 코딩 도구다. 이름만 보면 기능 하나처럼 보이지만, 실제로는 개인의 작성 습관뿐 아니라 팀의 리뷰 방식과 검증 흐름까지 건드릴 수 있는 제품군에 가깝다.
## 어떻게 작동하나
보통 IDE 확장 형태로 붙어서 현재 파일, 커서 주변 코드, 주석 같은 문맥을 바탕으로 제안을 만든다. 사용자는 그 제안을 그대로 받거나 일부만 고치고, 그 수정된 결과가 다시 다음 제안의 재료가 된다.
핵심은 브라우저에서 따로 AI를 여는 방식이 아니라, 코드를 쓰는 자리 바로 옆에 보조 기능이 끼어드는 구조라는 점이다. 그래서 흐름이 빠른 대신, 잘못된 제안도 손쉽게 받아들여질 수 있어서 검토와 테스트가 더 중요해진다.
## 왜 중요한가
실무에서는 단순히 타이핑 속도만 빨라지는 게 아니다. 반복 코드 작성, 테스트 초안 만들기, 낯선 코드 읽기, 주석이나 설명 정리 같은 자잘한 마찰이 줄어들면서 개발 병목이 다른 곳으로 옮겨간다.
기사나 제품 발표를 읽을 때도 이 구분이 중요해. 'Copilot 도입'이나 'Copilot 확장' 같은 말은 보통 추천 기능 하나를 깔았다는 뜻이 아니라, 팀이 AI를 IDE 안 기본 작업 흐름에 얼마나 깊게 넣었는지, 그리고 그에 맞춰 리뷰 기준·보안 정책·배포 전 검증 습관을 어떻게 바꾸는지까지 포함하는 경우가 많다.
## 주의해서 볼 점
Copilot이 제안한 코드는 그럴듯해 보여도 프로젝트 규칙, 보안 요구사항, 성능 제약을 정확히 반영하지 못할 때가 많아. 특히 테스트 없이 긴 코드를 한 번에 받아들이면 작성 시간은 줄어도 디버깅 비용이 뒤로 밀릴 수 있다.
팀 단위로 쓰면 더 중요한 건 개인 취향보다 운영 기준이다. 어떤 생성 코드를 그대로 받아도 되는지, 민감한 코드 문맥을 어디까지 다뤄도 되는지, 최종 책임과 리뷰 기준을 누가 지는지 먼저 정해두지 않으면 편해진 만큼 통제가 느슨해질 수 있다.
## 관련 용어
- [Claude Code](/ko/wiki/claude-code/)는 저장소를 직접 훑고 파일을 고치고 명령을 실행하는 작업 단위가 앞에 나온다. Copilot이 편집 중 옆에서 제안하는 보조자에 가깝다면, Claude Code는 할 일을 맡겨서 처리하는 코딩 에이전트 쪽에 더 가깝다.
- [Codex](/ko/wiki/codex/)는 원래 코드 생성 모델 계열 이름으로 많이 쓰였고, Copilot은 그 모델을 포함해 실제 사용자 경험으로 묶인 제품 이름이다. 둘을 같은 층위로 놓으면 모델 이야기와 도구 도입 이야기가 섞여 버린다.
- [Cursor](/ko/wiki/cursor/)는 AI를 중심에 둔 편집기 자체에 가깝고, Copilot은 기존 IDE에 붙는 보조 도구라는 출발점이 다르다. 그래서 Cursor는 개발 환경을 바꾸는 선택이고, Copilot은 익숙한 편집기를 유지한 채 AI 층을 얹는 선택으로 읽는 편이 정확하다.
- [Aider](/ko/wiki/aider/)는 CLI에서 여러 파일을 묶어 고치고 git 흐름과 연결하는 작업이 자연스럽다. Copilot은 타이핑 중 제안과 IDE 안 상호작용이 중심이라서, 둘은 같은 AI 코딩 도구여도 일하는 리듬이 다르다.
