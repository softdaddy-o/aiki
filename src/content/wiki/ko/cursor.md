## 한 줄 정의
Cursor는 AI를 깊게 붙인 코드 편집기야. 자연어로 작업을 지시하면 코드베이스를 이해한 뒤 파일 수정, 코드 생성, 변경 검토까지 한 흐름으로 이어 준다. 그래서 그냥 자동완성 툴로 보기보다, 개발자가 코드를 읽고 고치고 확인하는 방식을 재구성한 편집기라고 보는 쪽이 맞아.
## 어떻게 작동하나
프로젝트를 열면 Cursor는 코드베이스를 인덱싱해서 어떤 파일이 무엇을 하는지 찾기 쉬운 문맥을 만든다. 그다음 사용자가 채팅, 인라인 편집, 자동완성, 에이전트 같은 표면에서 요청하면 관련 파일과 규칙을 끌어와 코드를 제안하거나 직접 바꾸는 식으로 움직인다.
여기서 중요한 건 AI가 아무 데서나 제멋대로 쓰는 게 아니라는 점이야. `.cursor/rules` 같은 프로젝트 규칙, 사용자 규칙, `AGENTS.md`, 무시 파일 설정이 어떤 문맥을 넣고 어떤 파일을 건드릴지 좌우하고, 나온 결과는 diff review 화면에서 사람이 받아들이거나 거절할 수 있어. 터미널용 CLI도 있어서 편집기 밖 작업과 이어 붙이기 쉽다.
## 왜 중요한가
실무에서는 모델 이름 하나보다 개발 흐름을 어디서 묶느냐가 생산성 차이를 크게 만든다. Cursor는 코드 탐색, 수정 제안, 리뷰, 규칙 관리, 터미널 연결을 한 제품 안에 엮어 놓아서, 팀이 도입하면 프롬프트 습관만 바뀌는 게 아니라 PR 검토 방식과 저장소 규칙 관리까지 함께 바뀔 수 있어.
기사나 릴리스 노트에서 Cursor 얘기가 나오면 모델 출시 뉴스처럼 읽으면 자꾸 초점이 흐려져. Cursor는 배포 플랫폼 자체는 아니지만, 구현에서 검토까지의 마찰을 줄이는 쪽에 서 있어서 결과적으로 테스트 전 확인, 승인 흐름, 작업 분담 방식에 영향을 준다. 그래서 이 이름이 나오면 단순 기능 추가인지, 팀의 작업 절차를 바꾸는 제품 변화인지 구분해서 읽는 게 중요해.
## 주의해서 볼 점
Cursor를 모델 이름으로 읽으면 금방 헷갈려. Cursor의 핵심은 특정 모델 하나가 아니라, 여러 모델과 에이전트 기능을 편집기 경험 안에서 어떻게 연결하느냐에 있어.
또 자동으로 코드를 바꿔 준다고 해서 검토가 사라지는 건 아니야. 코드베이스 인덱싱 품질, 규칙 파일의 명확함, diff review 습관이 결과를 크게 좌우하고, 팀 규칙 없이 쓰면 같은 요청에도 산출물이 들쑥날쑥해질 수 있어. 결국 잘 쓰려면 '얼마나 똑똑하냐'보다 '어떤 범위까지 맡기고 어디서 사람이 끊어 보느냐'를 먼저 정해야 해.
## 관련 용어
- [Windsurf](/ko/wiki/windsurf/)는 Cursor처럼 AI 코딩 편집기라는 점에서 같은 층위에서 비교할 만해. 둘의 차이는 보통 모델 이름보다 에이전트 자율성, 편집기 통합도, 팀 규칙 관리 경험에서 갈린다.
- [Continue](/ko/wiki/continue/)는 기존 IDE에 붙여 쓰는 확장 성격이 더 강해. Cursor가 독립 편집기 경험을 전면에 내세운다면 Continue는 현재 쓰는 편집기와 모델 조합을 유지한 채 기능을 얹는 쪽에 가깝다.
- [Claude Code](/ko/wiki/claude-code/)는 터미널 중심 에이전트 흐름으로 읽는 편이 자연스러워. Cursor가 편집기와 변경 검토 화면을 중심에 둔다면 Claude Code는 셸 작업과 저장소 조작 감각이 더 앞에 나온다.
- [Codex](/ko/wiki/codex/)는 편집기 이름이라기보다 모델이나 코딩 에이전트 계열 이름으로 등장하는 경우가 많아. 그래서 Cursor와 나란히 놓고 볼 때는 '제품을 말하는지, 모델을 말하는지'부터 먼저 갈라서 이해해야 한다.
