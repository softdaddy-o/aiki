## 한 줄 정의
Prompt Engineering은 같은 AI 모델에 넣는 입력을 설계해서 원하는 출력 방향과 형식을 조정하는 기법이야. 핵심은 모델 자체를 바꾸는 게 아니라, 지시문과 배경정보와 예시와 형식 조건을 어떻게 묶어 넣을지 정하는 데 있어.
같은 모델이라도 질문을 어떻게 쓰는지에 따라 답의 정확도, 길이, 말투, 구조가 달라질 수 있어. 그래서 이 용어는 모델 교체나 재학습보다 입력 설계와 출력 제어에 더 가까운 말이야.
## 어떻게 작동하나
모델은 들어온 문장을 한 줄 질문으로만 보지 않고, 역할 지시, 작업 목표, 참고 정보, 예시 답안, 출력 형식 제한을 함께 읽고 다음 출력을 정해. 그래서 프롬프트를 설계할 때는 무엇을 해야 하는지, 무엇을 하지 말아야 하는지, 어떤 형식으로 내야 하는지를 분명하게 적는 게 중요해.
여기에는 예시를 몇 개 붙이는 방식, JSON처럼 결과 틀을 고정하는 방식, 긴 작업을 단계로 쪼개는 방식, 도구를 언제 호출할지 규칙을 주는 방식이 들어가. 이런 변화는 모델의 가중치를 다시 학습시키는 과정이 아니라, 현재 요청 안에서 모델이 참고할 조건을 조절하는 방식이야.
## 왜 중요한가
실제 사용에서는 같은 모델이라도 프롬프트 설계에 따라 결과 차이가 크게 난다. 입력이 모호하면 답이 흔들리고, 목표와 형식이 또렷하면 일관성과 재현성이 올라가.
이 구분을 알고 있으면 제품 설명이나 기사에서 성능이 좋아졌다고 할 때, 그 원인이 모델 교체인지 입력 설계 개선인지 더 정확하게 읽을 수 있어. 특히 문서 작성, 분류, 요약, 추출 같은 반복 작업에서는 프롬프트를 잘 짜는 것만으로도 작업 흐름을 꽤 안정시킬 수 있어.
## 주의해서 볼 점
프롬프트 엔지니어링이 만능 열쇠는 아니야. 비용과 지연시간 문제는 모델 선택이 더 직접적인 해법일 수 있고, 최신 사실을 다뤄야 하는 일은 검색이나 RAG 같은 바깥 정보 연결이 더 중요할 수 있어.
또 잘 만든 문장은 화려한 표현보다 경계가 분명한 지시에서 나온다. 목표가 불명확하거나 검증 규칙이 없으면 출력이 그럴듯해 보여도 실제 작업에는 맞지 않을 수 있어서, 프롬프트 설계와 결과 검증을 같이 봐야 해.
## 관련 용어
- [Chain-of-Thought (사고 연쇄)](/ko/wiki/chain-of-thought/)는 모델이 중간 추론 단계를 드러내게 만드는 프롬프트 기법이야. Prompt Engineering은 그보다 범위가 넓어서 역할 설정, 예시 제공, 형식 제한, 도구 사용 규칙처럼 입력 전체 설계를 다루고, Chain-of-Thought는 그중 추론 전개를 어떻게 끌어낼지에 더 초점이 있어.
